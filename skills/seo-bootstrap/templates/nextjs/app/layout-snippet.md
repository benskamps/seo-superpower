# `app/layout.tsx` — JSON-LD injection snippet

Reference snippet for the Next.js App Router root layout. Merge into your existing `app/layout.tsx`.

> **SAFETY NOTE:** The JSON-LD content here is generated server-side by our own helpers from a fixed schema — **never user input** — so the inline-HTML prop is safe in this exact use case. Do NOT pass user-controlled strings into these script tags. If your schema includes user-generated content (reviews, comments), sanitize first with DOMPurify or use a structured emitter that escapes correctly.

```tsx
import { organizationSchema, websiteSchema } from '@/lib/json-ld';

const orgLd = organizationSchema({
  name: 'Your Site',
  logoUrl: 'https://example.com/logo.png',
  sameAs: ['https://twitter.com/yoursite', 'https://www.linkedin.com/company/yoursite'],
});

const siteLd = websiteSchema({
  name: 'Your Site',
  searchUrlTemplate: 'https://example.com/search?q={search_term_string}',
});

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {/* JSON-LD: Organization + WebSite (entity-level, site-wide) */}
        {/* Two separate <script> tags is the recommended pattern — Google parses */}
        {/* each independently and one bad block won't poison the other. */}
        <script
          type="application/ld+json"
          dangerouslySetInnerHTML={{ __html: JSON.stringify(orgLd) }}
        />
        <script
          type="application/ld+json"
          dangerouslySetInnerHTML={{ __html: JSON.stringify(siteLd) }}
        />
        {children}
      </body>
    </html>
  );
}

// Don't forget metadataBase elsewhere in this file:
// export const metadata = { metadataBase: new URL('https://example.com'), ... }
```

## Why this pattern?

This is the [documented Next.js pattern for inlining JSON-LD](https://nextjs.org/docs/app/guides/json-ld). Alternatives:

- `<Script id="..." type="application/ld+json">{JSON.stringify(orgLd)}</Script>` — Next's `<Script>` component also works, but ships extra runtime.
- `next-seo` library — adds dependency, less control.

For static schema generated by your own helpers, the inline `<script>` approach is fine. For schema that includes user content, sanitize first.
