## Findings against the plugin

### [P0] No graceful degradation when MCPs aren't connected
`skills/seo-superpower/SKILL.md:12` says "fire `lighthouse-mcp` + `gsc-mcp` ping concurrently." `skills/auditing-technical-seo/SKILL.md:26` says "Call `lighthouse-mcp` for PSI." There's no documented fallback for "MCPs not configured yet." The user runs `/seo` after install but *before* `/seo-setup` finishes the OAuth wizard → confused state, no output.

**Fix:** Document and implement a no-MCP audit path. Static-resource checks (robots, sitemap, head tags, schema presence, viewport, security headers, response codes) need no auth and finish in ~30s. Output a partial `SEO_AUDIT.md` flagged "Lighthouse data missing — run /seo-setup for CWV."

### [P0] `/seo audit <url>` should work pre-`/seo-setup`
README sells "15 min from zero to value." Reality: `/seo-setup` is gate #1 (OAuth wizard, ~5 min, requires a Google account + clicks). A new user wants to *see something happen* before committing to OAuth.

**Fix:** Add `/seo audit <url>` argument path that runs the no-MCP fallback above. Lands in <60s with concrete findings. Demo-grade.

### [P1] Diagnostic doesn't reward "site is already in great shape"
The diagnostic flow always routes to a child skill that *changes things*. For a site like vibecrafting.ai, the right answer is "your foundation is solid — go work on content discovery." Skill currently has no "you pass the baseline, here's the 10% polish + the next growth move" branch.

**Fix:** In `seo-superpower` meta-skill, add a "Pass A" preamble: if 8+ baseline items check out (robots/sitemap/JSON-LD/OG/canonical/viewport/HTTPS/single-H1), output a congratulations line + skip to `finding-underserved-keywords` / `planning-topic-clusters` instead of bootstrapping.

### [P1] PSI doesn't require an MCP
Google's PageSpeed Insights API has a free tier with an API key (no OAuth, just a key from Cloud Console — same key the user already gets in `/seo-setup`). The plugin could ship a Python helper that hits PSI directly without standing up `lighthouse-mcp`.

**Fix:** Add `scripts/psi-quick.py <url>`. Reads `PSI_API_KEY` from env. Returns LCP/INP/CLS/TTFB at p75 from CrUX field data + lab fallback. Wire into the audit skill as a non-MCP path.

### [P1] No author-side dogfood path documented
As the plugin author, there's no documented way to run these skills against a target site without installing from the marketplace. I had to manually `Read` each `SKILL.md` and execute it by hand against vibecrafting.ai.

**Fix:** Document a "dogfood mode" section in `README.md` or new `DEVELOPING.md`: either a symlink recipe (`.claude/plugins/seo-superpower → ~/projects/seo-superpower`), or document `claude --plugin-dir ~/projects/seo-superpower` if that's a thing, or note "spin up a CC session in target repo with the local plugin installed via …"

### [P2] AI-bot robots.txt template would be high-leverage
vibecrafting.ai has the most thorough AI-bot stanza I've seen — exactly what `optimizing-for-generative-engines` should *generate* when missing. Right now the skill describes the policy choice; it doesn't ship a paste-ready template.

**Fix:** Add `templates/robots-ai-bots.txt` to the plugin. Generic version of vibecrafting.ai's stanza. Used by both `auditing-technical-seo` (when stanza missing) and `optimizing-for-generative-engines` (as the canonical recipe).

### [P2] `SEO_AUDIT.md` output path is undocumented in foreign repos
`auditing-technical-seo/SKILL.md:30` says "Write `SEO_AUDIT.md` to repo root." But when running this skill in vibecrafting's repo, that adds an audit doc to a user's source tree — they may not want that committed. No env var or arg controls the output path.

**Fix:** Document `SEO_AUDIT_OUTPUT` env var (default `./SEO_AUDIT.md`, override to e.g. `~/audits/vibecrafting-2026-05-12.md`). Add to `.gitignore` recommendation.

## What this dogfood tells us about the plugin

Strengths:
- The skills themselves are **substantive** — auditing-technical-seo has lifecycle awareness, output schema, common mistakes, false positives. Not vapor.
- The robots.txt + JSON-LD + sitemap recommendations match what a great real-world implementation (vibecrafting) actually does. The plugin's advice is correct.

Weaknesses:
- **Time to first signal is gated on OAuth.** Every demo requires `/seo-setup` to finish first. That's a churn cliff.
- **MCPs are assumed-present rather than gracefully optional.** Half the skills mention MCP calls without "if not connected, fall back to …" branches.
- **No "you're already good" branch.** Diagnostic is wired for sites that need fixing. Mature sites get awkward routing.

Net: the plugin is doc-rich and code-poor. Adding `scripts/psi-quick.py` + a no-MCP audit fallback + `/seo audit <url>` arg path would close the demo-grade gap.
